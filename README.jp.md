# self-correction-loop

タスクが完了したかどうかを、作業した本人ではなく**独立した検証者
(verifier)** が全基準を満たしたと確認するまで反復し続ける、
**自己修正ループ**として作業を構造化する Claude Code スキルです。

## なぜ作ったか

設計の動機となった、現在のモデルに関する 2 つの観察があります。

1. **モデルはフィードバックに対してヒルクライムすると着実に改善する。**
   チェック可能なゴール(テストスイート、スコアラー、ルーブリック)が
   あれば、モデルは実行→計測→調整→再実行を繰り返せます。難しいタスク
   では、このループはワンショットのプロンプティングを上回ります。
2. **モデルは自分の成果物の採点が苦手。** 作業を終えたモデルは成果物
   そのものではなく自分の意図に引きずられるため、自己批評は早めに
   勝利宣言をしがちです。*別のコンテキストウィンドウ*での採点 —
   ルーブリックと成果物だけを見て、作業者の思考過程は一切見ない
   検証者サブエージェント — の方が格段に信頼できます。

このスキルは両方のアイデアを再現可能な手順としてまとめ、さらにループが
誘発しがちな失敗モード、つまり指標のゲーミング(テストの弱体化、
スコアラーの改変、期待出力のハードコード)に対するガードレールを
備えています。また、作業があなたのチェックアウトの外に保たれます:
ループは使い捨ての git worktree(未コミット変更への dirty-state ガード
付き)の中で実行され、worktree が使えない環境では `.loop/` ディレクトリ
にフォールバックします。

## どんなときに使うか

**完了の定義が計測可能**で、一発では解けそうにないタスクに使います。

- 「テストが全部通るまでループして」
- 「このベンチマーク/スコアラーが 100 になるまでヒルクライムして」
- 「ルーブリックを満たすまでこのドキュメントを改善して」

ワンショットで済むタスクや、チェック可能なゴールのないタスクには
向きません。ループのオーバーヘッド(私たちのベンチマークでは実時間で
およそ 2〜4 倍、大半は検証者サブエージェント)は、客観的に収束できる
対象があって初めて元が取れます。

## ループの仕組み

```
0. 隔離
   使い捨ての git worktree の中で作業する — worktree が使えない場合は
   .loop/ ディレクトリ — ループがチェックアウトを汚さないようにする
        ↓
1. ルーブリック作成
   verbatim テンプレートを穴埋めする: 1 行の CHECK コマンド
   (exit 0 = 合格)+ 作業開始前に sha256 で固定した保護ファイル一覧
        ↓
2. ベースライン
   まず誠実に 1 回試行し、開始時点の計測値を EXPERIMENTS.md に記録する
        ↓
3. 反復
   変更は一度に 1 つ → 軽いチェックを実行 → 記録
   (CHANGE/WHY/RESULT/DECISION)。
   3 iteration 改善が無ければ微調整ではなく構造的な賭けに切り替える
        ↓
4. preflight + 検証
   scripts/preflight.sh が exit 0 になること(書式がパースでき、ガードが
   成立し、全 CHECK が通る)— その後、verbatim プロンプトテンプレートから
   フレッシュなコンテキストの検証者サブエージェントを起動する
        ↓
   FAIL の場合 → その evidence を入力として 3. に戻る
   PASS の場合 ↓
        ↓
5. 停止
   検証者の OVERALL: PASS が出たときのみ停止する。明示的な予算に達した
   場合は、暫定ベストと実験ログを報告して停止する
```

機械的な部分は散文ではなくスクリプトなので、弱いモデルでもドリフト
しません: `scripts/check_guards.sh` はルーブリックに記録された sha256
アンカーと保護ファイルを照合し、`scripts/preflight.sh` は書式検査と
CHECK コマンド検証で検証者の手前にゲートを置きます。ルーブリック・
実験ログ・検証者プロンプトの書式は `references/` の verbatim テンプレート
(プレースホルダのみ置換)で、設計理由は `references/rationale.md` に
あります。

各実行は監査証跡を残します: `RUBRIC.md`(ガードを含む「完了」の定義)と
`EXPERIMENTS.md`(全試行とその計測結果)、そして検証者による基準ごとの
判定です — これらはあなたのチェックアウトの外、worktree または cwd の
`.loop/`(`.git/info/exclude` で除外)に保たれます。成功時の報告には
ブランチ、diff 要約、マージ手順が含まれ、失敗時は worktree が検証用に
そのまま残ります。

スキルの完全な指示は
[self-correction-loop/SKILL.md](self-correction-loop/SKILL.md)
にあります。

## このリポジトリについて

このリポジトリはスキルの開発プロジェクトで、skill-creator の評価ループ
(テストプロンプトをスキルあり/なしで実行し、アサーションで採点し、
レビューして改訂する)で反復開発しています。

- `self-correction-loop/` — スキル本体
- `evals/` — ベンチマーク用のテストケースとフィクスチャ
- `self-correction-loop-workspace/`(gitignore 対象)— 生成された実行結果

開発ワークフローとベンチマークコマンドは [CLAUDE.md](CLAUDE.md) を
参照してください。
