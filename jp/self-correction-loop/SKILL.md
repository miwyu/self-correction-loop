# 自己修正ループ (Self-Correction Loop)

作業をループとして構造化する: 試行 → 計測 → 改善 → 繰り返し。そして
**独立した検証者**がすべての基準を満たしたと確認したときにのみ停止する。

独立性が重要な理由: 何かに取り組んだ後のあなたは、成果物そのものでは
なく自分の意図に引きずられており、自己批評は実際には終わっていない
作業を合格させがちである。フレッシュなコンテキストウィンドウを持つ
検証者サブエージェントは、ルーブリックと成果物だけを読み — あなたの
思考過程は決して読まない — その判定は実際にそこにあるものを反映する。

## ステップ 1: 作業を始める前にルーブリックを書く

ユーザーのゴールを、作業ディレクトリのルーブリックファイル `RUBRIC.md`
に変換する。すべての基準は客観的にチェック可能でなければならない —
コマンドの実行、または成果物の検査によって。曖昧な基準(「コードが
きれい」)は検証者にチェックするものを与えず、ループを早期に止めて
しまう。

- ゴールにすでにチェック可能なシグナル(テストスイート、スコアラー
  スクリプト、lint コマンド)が付いているなら、そのコマンドが
  ルーブリックの中核になる。正確なコマンドと要求される結果を記録する。
- 解くのではなくゲーミングによってゴールが満たされうる経路に対して、
  **ガード基準**を追加する:「テストファイルは改変されていない」
  「スコアラースクリプトは編集されていない」「期待出力はハードコード
  されていない」。合格へのループの圧力は、まさにこうした近道を生む
  圧力でもあるので、最初から名指ししておく。
- ゴールの一部が本質的に定性的なら、フレッシュな読み手が成果物だけから
  イエス/ノーで答えられるように言い換える(例:「`--help` に現れる
  すべての CLI フラグに README 内の説明がある」)。

## ステップ 2: ベースラインを確立する

まず誠実に 1 回試行し、計測可能なチェックを実行して結果を記録する。
ベースラインは、後の変更が本当に改善なのかを教えてくれ、ユーザーに
出発点がどこだったかを伝える。

## ステップ 3: 反復する

各反復で:

1. **変更は 1 つだけ。** 最も効果が見込めるアイデアを選ぶ。一度に
   複数の変更を入れてスコアが動いた場合、なぜ動いたのか分からなくなる。
2. **軽いチェックは自分で実行する**(テストコマンド、スコアラー)。
   すでに失敗すると分かっている試行に検証者の実行を浪費しない。
3. **実験を記録する**: `EXPERIMENTS.md` に、何を試したか、なぜか、
   計測結果、採用か差し戻しか。このログは失敗した実験の繰り返しを防ぎ、
   ユーザーにとって実行の窓になる。

次に何を試すか選ぶとき: スカラーな微調整(定数の調整、リネーム、
並べ替え)は安価だがすぐに頭打ちになる。進捗が停滞したら、構造的な
賭けをする — 別のアルゴリズム、別の表現、問題の捉え直し — たとえ
一時的にスコアが後退しても。ログ上の停滞こそが、チューニングから
再構築へ切り替える合図である。

## ステップ 4: 独立に検証する

すべての基準が合格すると確信したら、Agent ツールで検証者サブ
エージェントを起動する。渡すのはルーブリックと成果物のパスだけ —
何をしたか、なぜ合格するはずかというあなたの要約は渡さない。
使えるテンプレート:

> `<path>/RUBRIC.md` のルーブリックを読んでください。各基準について、
> `<path>` 内の実際の成果物を確認し(基準がコマンドを指定している
> 場合は実行し)、判定を返してください: 基準ごとに PASS または FAIL、
> それぞれ具体的な根拠 1 行(コマンド出力、ファイル内容、行番号)
> 付きで。懐疑的であってください: あなたの仕事は成立して*いない*基準を
> 見つけることです。ファイルは一切変更しないでください。最後に総合
> 判定を出してください: すべての基準が合格した場合に限り PASS。

検証者がある基準を FAIL にしたら、その根拠を次の反復の入力として扱い、
ステップ 3 に戻る。判定に反論したり自分で再採点したりしない — 基準が
「未達成」ではなく「そもそも誤っている」ように見える場合は、黙って
落とすのではなくユーザーに報告する。

## ステップ 5: 停止する

- **成功:** 検証者がすべての基準に合格を出した。最終判定と、
  ベースラインから最終結果への改善を添えて報告する。
- **予算:** ユーザーが反復回数や時間の予算を与えたなら、それに従う。
  与えられておらず、連続する複数の反復(少なくとも 1 回の構造的な
  賭けを含む)で進捗がない場合は停止し、それまでのベストの結果、
  実験ログ、追加予算があれば次に試すことを報告する。

このセッションでの検証者の合格判定なしに、決して成功を主張しない。
