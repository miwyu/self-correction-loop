# 自己修正ループ (Self-Correction Loop)

ループ: 試行 → 計測 → 改善。独立した検証者がすべての基準を確認したとき
にのみ停止する。手順は以下のとおり。各規則の設計理由は
`references/rationale.md` にある(読むのは興味があるときで、ループの実行
には不要)。

`<skill_dir>` = この SKILL.md があるディレクトリ。
`<ws>` = ステップ 0 で選ぶワークスペースルート。ループの成果物はすべて
`<ws>/.loop/` に置く。CHECK コマンドは必ず `<ws>` から実行する。

## MUST / NEVER

- **MUST M1:** 成功を宣言できるのは、このセッション内で `preflight.sh` が
  exit 0 になり、**そのあとで**検証者サブエージェントの報告が
  `OVERALL: PASS` で終わったときだけ — この順序で、どちらも成果物の現在の
  状態に対して。
- **MUST M2:** `<skill_dir>/references/` の 3 つのテンプレートは verbatim に
  コピーし、プレースホルダのみ置換する。
- **MUST M3:** 1 iteration につき変更は 1 つ。そしてすべての iteration を、
  先へ進む前に `EXPERIMENTS.md` に記録する。
- **NEVER N1:** RUBRIC.md の `## Protected files` に列挙したファイルを
  変更しない。
- **NEVER N2:** `.loop/`、RUBRIC.md、EXPERIMENTS.md を削除・「掃除」しない —
  これらはユーザーの監査証跡であり、レビュー後に削除するのはユーザーである。
- **NEVER N3:** テスト・スコアラーの期待出力やテストケースを RUBRIC.md や
  解答にコピーしない。
- **NEVER N4:** 検証者プロンプトをパラフレーズ・拡張・要約混入しない。
- **NEVER N5:** ユーザーが求めていないリポジトリを `git init` しない。
  ユーザーのブランチにコミットしない。
- **NEVER N6:** 検証者が `OVERALL: PASS` を返したあとに rubric のチェックを
  再実行したり成果物を再点検したりしない — 判定は最終であり、すでに
  コンテキストにある。PASS 後に許されるツール使用は、ステップ 5 の
  hand-back 自体が必要とする git コマンド(diff 要約、status 確認)のみ。

## ステップ 0: ワークスペースを選ぶ

1. cwd が git リポジトリで**ない**場合(または worktree 隔離そのものが
   不可能・重すぎる場合 — 例: フレッシュチェックアウトでは再現できない
   重い環境): `<ws>` = cwd。ステップ 1 へ。
2. cwd が git リポジトリ**である**場合: 使い捨て worktree で作業する —
   Claude Code なら EnterWorktree、なければ
   `git worktree add ../<repo>-loop -b <branch>`(リポジトリの隣に置く)。
   `<ws>` = その worktree。
   - **dirty-state ガード:** worktree はコミット済み状態から始まる。まず
     `git status` を確認し、タスクが触る未コミットファイルは worktree に
     コピーする(WIP コミットよりコピーを優先 — ユーザーのブランチと
     インデックスに触れない)。
   - タスクの計測コマンドを早めに一度実行する。worktree はフレッシュ
     チェックアウトである(`.env` も依存も無い)。最小限のセットアップを
     行い、重すぎるなら規則 1 にフォールバックする。
   - **ベースライン照合:** ステップ 2 のベースラインがユーザーの説明と
     矛盾する(「失敗するはず」のテストが通る)なら、まず stale な基底を
     疑う: 元チェックアウトの未コミット変更を調べる。

## ステップ 1: ループワークスペースの設営(タスクの作業を始める前に)

1. 1 つのコマンドブロックで(`.loop/` が既に存在するなら上書きせず停止して
   報告する。git リポジトリ外では exclude の行はスキップ):
   ```bash
   test ! -e <ws>/.loop && mkdir -p <ws>/.loop \
     && cp -R <skill_dir>/scripts <ws>/.loop/scripts \
     && echo '.loop/' >> "$(git -C <ws> rev-parse --path-format=absolute --git-common-dir)/info/exclude" \
     ; shasum -a 256 <ws>/<保護対象の各ファイル>
   ```
2. `<skill_dir>/references/rubric-template.md` と
   `<skill_dir>/references/experiments-template.md` を(両方いま)読み、
   `<ws>/.loop/RUBRIC.md` と `<ws>/.loop/EXPERIMENTS.md` を作る。
   基準は客観的にチェック可能でなければならない。ユーザーが述べたコマンドは
   verbatim に CHECK 行へ。ゴールがゲーミングで満たされうる経路(テスト
   改変、スコアラー編集、期待出力のハードコード)にはガード基準を追加する:
   該当ファイルを `## Protected files` に、手順 1 で記録した sha256 と
   ともに — **作業を始める前に** — 列挙する。CHECK は必ず 1 行コマンドに
   保つ(必要なら `grep -q` / `diff` にパイプ)。1 行で書けないチェックは
   MANUAL 基準にする — ヘルパーのチェッカーファイルを書かない。ループの
   ファイルは RUBRIC.md、EXPERIMENTS.md、コピーした `scripts/` だけである。

## ステップ 2: ベースライン

何かを変える前に、中核の CHECK コマンドを一度実行し、結果を verbatim に
EXPERIMENTS.md の `## Baseline` に記録する。ベースラインは以後の結果の
比較対象であり、ステップ 0 のガードで照合する対象でもある。

## ステップ 3: 反復

各 iteration:

1. **1 つ**だけ変える(M3)— 最も効くと予想する変更を。
2. 関係する CHECK コマンドを自分で回す(安価。サブエージェント不要)。
3. EXPERIMENTS.md に `## Iteration N` ブロックを追記する
   (CHANGE / WHY / RESULT / DECISION — フィールドはテンプレートが固定)。
4. worktree では: iteration をコミットする — タスクが変更すべきファイル
   のみ(`.loop/` は exclude 済みで untracked のまま)。

**プラトー規則:** 指標が 3 iteration 連続で改善しなかったら、次の
iteration は構造的変更(別アルゴリズム・別表現・問題の別定式化)で
なければならない。微調整の続行は不可。**予算:** ユーザーが iteration 数や
時間の予算を与えたならそれに従う。なければ、構造的変更を少なくとも 1 回
含む 3 iteration 連続の非改善で停止して報告する。

## ステップ 4: preflight、そして検証

1. `bash <ws>/.loop/scripts/preflight.sh <ws>/.loop` を実行する。
   exit 0 が必須。失敗したら直してステップ 3 へ戻る。preflight が失敗して
   いる間は検証者を起動しない。
2. `<skill_dir>/references/verifier-prompt.md` を読む。`{{LOOP_DIR}}` を
   `<ws>/.loop` の絶対パスで置換し — 許される編集はこれだけ(N4)—
   そのテキストで検証者サブエージェント(Agent ツール)を起動する。
   自分の要約・diff・理由づけを決して含めない。
3. 最終行 `OVERALL: PASS` → ステップ 5 へ。いずれかが FAIL → その
   evidence を次の iteration の入力としてステップ 3 へ戻る。判定を再採点
   したり反論したりしない。基準そのものが「未達」ではなく「誤り」に見える
   なら、黙って落とさずユーザーに提起する。

## ステップ 5: 報告

検証者の PASS は最終である — その後にチェックを再実行したりファイルを
再点検したりしない。報告を書いて終了する。

- **成功**(M1 充足): 検証者の判定、ベースライン → 最終結果、使った
  iteration 数を報告する。
  - worktree モード: 修正だけを引き渡す — ブランチ名、diff の要約、正確な
    着地手順(マージ、または `git checkout <branch> -- <files>`)。既定は
    手順の提示である。自分で適用するのは、ユーザーの作業を失うことが
    証明可能に不可能な場合(チェックアウトがクリーン、または diff が
    ユーザーの未コミット内容を包含する場合)だけで、どちらを行ったかを
    明言する。レビュー後の掃除コマンドを添える:
    `git worktree remove --force <path>`(`--force` が untracked の
    `.loop/` を消す)、その後ブランチ削除。
  - フォールバックモード: `<ws>/.loop/`(rubric + 実験ログ)が残っている
    ことと削除方法を 1 行で伝える。自分では消さない(N2)。
- **予算切れ / 停止:** best-so-far、実験ログ、次に試すことを報告する。
  ワークスペースは worktree も含めそのまま残し、パスを報告する。
