# 設計理由 (Design rationale)

ループがこの形をしている理由。理由が知りたいときに読む。SKILL.md は
意図的に手順だけを載せている。

## なぜ独立した検証者か

何かに取り組んだ後のあなたは、成果物そのものではなく自分の意図に
引きずられており、自己批評は終わっていない作業を合格させがちである。
フレッシュなコンテキストを持つ検証者サブエージェントはルーブリックと
成果物だけを読み — あなたの理由づけは決して読まない — その判定は実際に
そこにあるものを反映する。検証者プロンプトが verbatim テンプレートである
のも同じ理由だ: パラフレーズは基準に対するあなたの解釈を紛れ込ませる機会
であり(弱いモデルでの実測: 書き直されたプロンプトは 8 run 中 3 で
overall-PASS 規則を落とし、8 run 中 1 でガード基準を弱体化した)、
作業の要約を検証者に渡さないのも同じ理由である。

## なぜガードに sha256 アンカーか

「テストファイルは改変されていない」は散文としてはチェック不能である:
比較対象が無ければ、検証者は弱い代理指標(「skip マーカーが無い、見た目は
正常」)に落ちて合格させてしまう。作業開始前にハッシュを記録すれば、
ガードは機械的な比較になる — `check_guards.sh` は一致するかしないかだけだ。
ガードが存在するのは、合格へのループの圧力こそがテスト改変・スコアラー
編集・期待出力ハードコードを生む圧力だからであり、保護ファイルを先に
名指しすることが近道を可視のままにする。

## なぜ期待出力をルーブリックに入れないか

スコアラーのケース表を RUBRIC.md にコピーすることは、ルックアップテーブル
を解答の一歩手前に置き、検証者を「スコアラーを実行する」のではなく
「コピーと照合する」方向へ偏らせる。ルーブリックが記録するのはコマンドと
要求される結果であり、テストデータそのものでは決してない。

## なぜ検証者の前に preflight か

自分の CHECK コマンドすら通らない試行に検証者を使うのはサブエージェントの
無駄遣いである。機械的なゲート(成果物の存在、書式のパース、ガードの成立、
全 CHECK の exit 0)がそれを濾し取り、検証者はスクリプトにできないこと —
MANUAL 基準への懐疑的な判断と独立した再実行 — に集中する。

## なぜ 1 iteration に 1 変更か

複数のことを同時に変えてスコアが動いたら、なぜ動いたのか分からず、
実験ログに知識が蓄積しなくなる。スカラー微調整はすぐ頭打ちになる。
「3 iteration 改善なし」の閾値は、死んだチューニングを挽き続ける代わりに
構造的な賭け(別アルゴリズム・別表現)への切り替えを強制する。

## なぜ worktree か、なぜ成果物を untracked のままにするか

ループは投機的な編集を大量に行う。使い捨ての worktree はそれらをユーザー
のブランチとインデックスから完全に隔離する。worktree はコミット済み状態
から始まり、未コミットの変更は持ち込まれない — dirty-state ガードがある
のはそのためだ: 対象の未コミットファイルをコピーしなければ、stale な
バージョンを「修正」してしまう(ベースライン矛盾チェックはまさにそれを
捕まえるために存在する)。RUBRIC.md と EXPERIMENTS.md はプロセスの
成果物であってプロダクトではない: `.loop/` に置かれ、`.git/info/exclude`
(ユーザーの .gitignore には決して触れない)で除外されるので、後のどんな
マージもそれらをユーザーのブランチへ運ばない。これらはユーザーの監査証跡
でもある — 実行後に消すことは、何が試されたかの唯一の証拠を破壊する。
掃除はユーザーに委ねる。

## なぜ成功に「このセッションの OVERALL: PASS」が要るか

stale な判定(以前の試行のもの)や自己採点の「全チェック合格」は、ループ
が早期に止まる典型経路である。譲れない規則はただひとつ: 検証されたのは
成果物の最後の状態だ、ということである。
